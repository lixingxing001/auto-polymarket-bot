# Task Plan: BTC 5分钟涨跌交易机器人

## Goal
搭建一个可验证、可回测、可先纸面交易再逐步上线的 BTC 5 分钟方向交易机器人方案与最小代码骨架。

- [x] Phase 2: 调研当前可用市场、数据与执行约束
- [x] Phase 3: 设计系统架构与实验路线
- [x] Phase 4: 搭建最小工程骨架
- [x] Phase 5: 复核风险、给出下一步
- [x] Phase 6: 接入 Polymarket 当前市场发现与订单簿读取
- [x] Phase 7: 构建纸面交易主循环
- [x] Phase 8: 构建历史标签数据集与基线评估
- [x] Phase 9: 构建纸面信号结算与前向收益评估
- [x] Phase 10: 接入可执行价格回测或更强特征
- [x] Phase 11: 训练可复现模型并做时间切分评估
- [x] Phase 12: 做基于真实成交的可执行价格回测
- [x] Phase 13: 接入 Polymarket 历史价格并训练 market-aware 模型
- [x] Phase 14: 用 market-aware 模型重跑可执行价格回测
- [x] Phase 15: 扩大 walk-forward 样本并记录自有盘口快照
- [x] Phase 16: 接入快照驱动回测
- [ ] Phase 17: 积累高频前向样本并切换 WebSocket 数据源

## Key Questions
1. 用户真正要做的是二元预测市场交易，还是交易所上的 BTC 短周期策略
2. 5 分钟方向信号在扣除费率、滑点、延迟后是否仍然可交易
3. 哪些模块必须先验证，哪些模块可以后置

## Decisions Made
- 当前仓库为空，从零开始更合适。
- 先做研究与纸面交易骨架，再考虑实盘执行，原因是 5 分钟级别最容易被噪声和交易成本吞掉。
- 默认目标先按 Polymarket 的 BTC 5 分钟 Up/Down 市场设计，因为仓库名和用户语境都指向 Polymarket。
- 当前实现必须基于 CLOB V2，不能照抄旧版 SDK 方案。
- 既然目标已经明确，后续只围绕 Polymarket 的 5 分钟 BTC Up/Down 市场展开。
- 当前窗口 slug 可由 UTC 时间向下取整到 5 分钟后拼接生成，优先走确定性发现路径。
- 历史验证先做“方向信号评估”，先看样本外是否有可复现信息，再补充可执行价格做收益回测。
- 历史样本统一在开窗后 60 秒决策，避免把整根 5 分钟 K 线的未来信息偷塞进特征里。
- 80% 暂时只作为远期目标，当前阶段优先优化样本外泛化与高置信度交易质量。
- 由于当前文档没有暴露历史订单簿快照查询，先用市场真实成交构造保守的“可执行价格代理”回测，再逐步积累自有盘口快照。
- 后续要把“样本扩容”和“盘口快照采集”拆成两条独立管线，避免研究与采集互相阻塞。
